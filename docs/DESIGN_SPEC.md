# Cocoro Influencer - System Architecture & Development Flow

## 1. プロジェクトの基本方針
本プロジェクトは、**「1リポジトリ・完全モジュール分割」**を原則とします。
複数の開発メンバーが同時に開発を進められるよう、各機能（URL解析、画像生成、リップシンク等）を独立したディレクトリに分離し、お互いのコードに干渉せず実装・テストが可能な構造に再編します。

## 2. システムディレクトリ構成案
以下の構造により、関心事を完全に分離します。

```text
cocoro-influencer/ (or hybrid-avatar-pipeline/)
├── docs/
│   └── DESIGN_SPEC.md           # 本設計書・API仕様書など
├── src/
│   ├── modules/
│   │   ├── analyzer/            # 【担当A】URL解析・参考動画分析
│   │   ├── script_gen/          # 【担当B】AI台本自動生成 (LLM)
│   │   ├── avatar_gen/          # 【担当C】キャラ顔固定・ベースキャラ画像生成
│   │   ├── voice_gen/           # 【担当D】音声合成・声生成 (TTS)
│   │   ├── video_gen/           # 【担当E】動画生成・シネマティック映像生成 (Kling I2V等)
│   │   ├── lipsync/             # 【担当F】リップシンク合成・トーキングヘッド生成 (SyncLabs等)
│   │   └── compositor/          # 【担当G】最終レンダリング・シーン結合・テロップ付与
│   ├── api/                     # FastAPI サーバー・ルーティング群
│   └── cli.py                   # コマンドラインツール (cocoro-inf相当)
├── ui/                          # Webフロントエンド (index.html, JS, CSS)
├── assets/                      # プリセット画像、BGM、フォント等
├── tests/                       # 各モジュールの単体テスト
├── config/                      # 環境変数 (.env) および共通設定
└── requirements.txt             # 依存関係
```

## 3. モジュール別 開発スコープとI/O（入力/出力）
各チームメンバーは、自分のモジュールが「何を受け取り」「何を返すか」だけを意識して開発します。

### 3.1. `analyzer` (URL解析)
*   **Input**: YouTube・TikTok等のURL、または実動画ファイル
*   **Output**: JSON形式のシーンリスト、抽出された台本テキスト

### 3.2. `script_gen` (台本生成)
*   **Input**: ユーザーが入力したテーマ・キーワード
*   **Output**: AIによる最適化テキスト、タイムラインごとの分割台本

### 3.3. `avatar_gen` (キャラ固定・画像生成)
*   **Input**: キャラクターのプロンプト、またはLoRA等の設定パラメータ
*   **Output**: 生成されたベースアバター画像パス (`.png`)

### 3.4. `voice_gen` (声生成)
*   **Input**: 台本テキスト、話者設定
*   **Output**: 生成された音声データパス (`.wav`)

### 3.5. `video_gen` (動画生成)
*   **Input**: ベース画像パス、演出プロンプト (Kling API等への送信)
*   **Output**: シネマティックな無音の動画パス (`.mp4`)

### 3.6. `lipsync` (リップシンク合成)
*   **Input**: `video_gen`からの動画パス、`voice_gen`からの音声パス
*   **Output**: 口の動きが完全に同期したトーキング動画パス (`.mp4`)

### 3.7. `compositor` (最終レンダリング)
*   **Input**: 全シーンの動画クリップ配列、BGM設定
*   **Output**: SRT字幕焼き付け済みの最終結合動画 (`final.mp4`)

## 4. チーム開発の進め方（フロー）
1.  **インターフェース定義（規約）**: 各モジュールの呼び出し方（例: `lipsync.generate(video_path, audio_path)`）を先にチーム内で決定します。
2.  **モック（ダミー）実装**: 初期段階では、実際にAIを動かさず、ダミーの動画や画像を返すだけの関数を実装し、全パイプラインが通るか確認します。
3.  **個別最適化**: 各担当者は、自分のモジュール配下のスクリプトのみをいじり、精度向上・API連携・重いGPU処理の実装に集中します。
4.  **単体テスト**: 各自が `tests/` ディレクトリ内に自分のモジュール用テストを書き、単独で動くことを保証します。
5.  **結合**: `src/api/` (Web UI用) または `src/cli.py` (バッチ処理用) 経由で全機能を連結し、最終アウトプットを得ます。

---
この設計により「動画生成チームはKlingの調整に集中」「AIリサーチャーはOllama 120Bを使った台本生成ループの改良に集中」といった並行開発が完全にスムーズに行えます。
